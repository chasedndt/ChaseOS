# ChaseOS MCP Audit Policy

> Defines the exact audit record fields, write location, timing, and failure policy for every MCP surface class and active V2 invocation surface.
>
> This document removes all ambiguity about what is logged, when, and what happens when logging fails.
> Pass 4 implemented `audit/logger.py` for V1. Pass 6B extends the policy for `workflow.invoke_bounded`. Pass 6C documents audit reconciliation and duplicate live-output denial for operator use.

---

## Governing Principle

**Every MCP request produces an audit record — no exceptions.**

The only legitimate reason an audit record does not exist for a request is:
1. The request failed before permission envelope resolution (malformed input — no audit)
2. The audit write failed and the request was fail-open (resource reads, prompts, validate/diff_preview, approval_request.create after artifact write) - in which case the failed write attempt itself is logged internally
3. The audit write failed after AOR returned for `workflow.invoke_bounded` - in which case the caller receives `system_error(workflow_invocation_audit_failed)` and no success response is permitted

An MCP surface that produces no audit trail is a governance violation, not a design choice.

Audit ownership is held by the server/envelope layer only:
- Surface handlers return structured audit metadata.
- Surface handlers do not write audit files.
- Surface handlers do not call `MCPAuditLogger`.
- `MCPAuditLogger` is called only by `server.py` / the shared request envelope.

---

## Audit Record Location

All audit records land at:

```
07_LOGS/Agent-Activity/
```

This is the same target used by AOR engine audit records. Both share the same directory intentionally — it is the canonical agent activity log for the vault.

**Why not a separate MCP-specific log dir?** A separate dir would fragment the audit trail. Operators reviewing `07_LOGS/Agent-Activity/` should see all agent activity — AOR workflow execution and MCP surface requests — in one place.

---

## Audit Record Naming Convention

```
{timestamp}__mcp__{surface_id}__{request_id[:8]}.json
```

**Examples:**
```
20260420-113042__mcp__chaseos.current_truth__a1b2c3d4.json
20260420-113115__mcp__proposal.submit__b2c3d4e5.json
20260420-113200__mcp__handoff.runtime_draft_frame__c3d4e5f6.json
```

**Rules:**
- `timestamp` — UTC, format `YYYYMMDD-HHMMSS`
- `mcp` — literal namespace prefix distinguishing from AOR records
- `surface_id` — the MCP surface name (resource URI, tool name, or prompt name); dots preserved, no sanitization
- `request_id[:8]` — first 8 hex characters of the request's UUID4 request ID

This naming is consistent with the AOR convention (`{timestamp}__{workflow_id}__{audit_id[:8]}.json`) while preserving the `mcp` namespace marker for MCP envelope records.

## Audit Record Schema

Every audit record is a JSON file with this fixed schema:

```json
{
  "schema_version": "1.0",
  "request_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "recorded_at": "2026-04-20T11:30:42Z",
  "surface_id": "chaseos.current_truth",
  "surface_class": "resource",
  "runtime_id": "openclaw",
  "trust_tier": 2,
  "safety_mode": "read_plus_proposal",
  "outcome": "success",
  "outcome_detail": null,
  "files_read": ["00_HOME/Now.md"],
  "files_written": [],
  "error_code": null,
  "error_message": null
}
```

### Field Definitions

| Field | Type | Required | Notes |
|---|---|---|---|
| `schema_version` | string | yes | Always `"1.0"` for V1 |
| `request_id` | string (UUID4) | yes | Full UUID from the MCP request; generated by server if absent |
| `recorded_at` | string (ISO 8601 UTC) | yes | Timestamp of audit write, not request receipt |
| `surface_id` | string | yes | Exact MCP surface name (e.g. `chaseos.current_truth`, `proposal.submit`) |
| `surface_class` | string enum | yes | `"resource"`, `"tool"`, or `"prompt"` |
| `runtime_id` | string | yes | Identifying runtime from permission envelope |
| `trust_tier` | integer | yes | Integer tier (1, 2, or 3) from config at time of request |
| `safety_mode` | string | yes | Active safety mode for this session |
| `outcome` | string enum | yes | `"success"`, `"error"`, or `"partial"` |
| `outcome_detail` | string or null | no | Optional human-readable note on why outcome was partial/error |
| `files_read` | array of strings | yes | Vault-relative paths of all files read (empty array if none) |
| `files_written` | array of strings | yes | Vault-relative paths of all files written (empty array if none) |
| `error_code` | string or null | yes | MCP error code if outcome is error; null otherwise |
| `error_message` | string or null | yes | Error message if outcome is error; null otherwise |

**`files_written` for proposal tools:**
- `proposal.submit`: populated with the staging artifact path (`.chaseos/mcp-proposals/{name}.json`)
- `proposal.validate`, `proposal.diff_preview`: always empty (no writes)
- `approval_request.create`: populated with the `07_LOGS/Operator-Briefs/` artifact path
- `workflow.invoke_bounded`: populated with AOR-returned relative artifact paths only; MCP does not include generated brief content

## Audit Timing

**Audit writes are synchronous and happen before the response is returned.**

```
Handler completes → envelope writes audit → response sent to caller
```

This is non-negotiable for proposal.submit (see fail-closed policy below). For fail-open surfaces, the same timing applies — the audit write is attempted before response, and on failure, the response is sent anyway.

**No deferred or batched audit writes.** The caller may observe slightly elevated latency on `proposal.submit` and `workflow.invoke_bounded` because the audit write must complete before the success response is returned.

---

## Fail-Open vs Fail-Closed Policy by Surface Class

This is the frozen ruling. There is no ambiguity.

### Resources (all 9 V1 resource reads) — FAIL-OPEN

**Audit failure behavior:** Log the failure to the server's internal error log. Continue and return the resource response to the caller. Do not surface the audit failure to the caller.

**Rationale:** Resource reads are read-only. An audit failure does not create a governance risk because no vault state changed. Blocking a resource read due to an audit write failure would make the server unreliable for legitimate read operations (e.g. a full 07_LOGS/ directory or permissions issue on a specific subdirectory). The audit failure is a separate concern.

**Exception:** If audit writes fail for more than N consecutive requests (N = 10, implementation decision for Pass 4), the server should warn the operator at the next request that audit logging is degraded. This warning appears in the response metadata, not as an error response.

---

### proposal.submit — FAIL-CLOSED

**Audit failure behavior:** If the audit write fails after staging succeeds → rollback the staging artifact (delete it from `.chaseos/mcp-proposals/`) → return `system_error(audit_required)` to the caller.

**Rationale:** `proposal.submit` is the only V1 tool that creates a durable artifact with governance implications. A staged proposal with no audit record violates the core invariant: every MCP write action must have a trail. The audit record is not optional for proposal.submit — it is a governance requirement. Rolling back staging preserves the invariant: if no audit record exists, no staged artifact exists.

**Rollback procedure:**
1. `stage()` succeeds — artifact exists at `.chaseos/mcp-proposals/{name}.json`
2. `log()` is called — write fails
3. Server calls `staging_store.rollback(proposal_id)` — deletes the staging artifact
4. Returns `system_error(audit_required)` — caller sees a clean failure

If the rollback itself fails (e.g. the staging artifact cannot be deleted), the server returns `system_error(audit_and_rollback_failed)` and logs the inconsistency to the internal error log. This is a non-recoverable state in V1 — the operator must manually inspect `.chaseos/mcp-proposals/`.

---

### proposal.validate — FAIL-OPEN

**Audit failure behavior:** Log the failure internally. Return the validation result to the caller.

**Rationale:** `proposal.validate` makes no writes and no state changes. Its output is a validation report. An audit failure does not create governance risk. Blocking validation due to audit failure would prevent the caller from getting a validation result without serving any safety purpose.

---

### proposal.diff_preview — FAIL-OPEN

**Audit failure behavior:** Log the failure internally. Return the diff result to the caller.

**Rationale:** Same as `proposal.validate`. No writes, no state changes. The diff preview is informational.

---

### approval_request.create — SPLIT: FAIL-CLOSED for artifact write, FAIL-OPEN for audit

This surface has two distinct writes:
1. **Artifact write** to `07_LOGS/Operator-Briefs/` — **FAIL-CLOSED**
2. **Audit write** to `07_LOGS/Agent-Activity/` — **FAIL-OPEN** (after artifact write)

**Why fail-closed for artifact?** The approval request artifact is the primary deliverable of this tool. A success response without the artifact would be a lie — the operator would believe a review request was delivered when it was not. The artifact write must succeed before success is returned.

**Why fail-open for audit?** The artifact already exists. The state change (the approval request being delivered to the operator) is real and cannot be rolled back. An audit write failure after a successful artifact write should not force the caller to retry and potentially create a duplicate approval request artifact. The failure is logged internally.

**Sequence for approval_request.create:**
```
1. Validate proposal_id exists in staging         → if not: input_error, envelope writes error audit
2. Write artifact to 07_LOGS/Operator-Briefs/     → if fails: system_error(artifact_write_failed), envelope writes error audit
3. Envelope writes audit record with status=created → if fails: log internally, return success (artifact exists)
4. Return success response
```

---

### workflow.invoke_bounded - FAIL-CLOSED after AOR returns

This active V2 surface has two audit streams:
1. **AOR audit** to `07_LOGS/Agent-Activity/` - owned by AOR during `run_workflow()`
2. **MCP audit** to `07_LOGS/Agent-Activity/` - owned by the MCP server/envelope after the handler returns

**Audit failure behavior:** If AOR has returned and the MCP audit write fails, the server returns `system_error(workflow_invocation_audit_failed)`. It does not re-run AOR, does not attempt canonical rollback, and does not return success.

**Rationale:** The workflow may have already produced an Operator-Briefs artifact through AOR Stage 7. Returning success without an MCP envelope audit would break the governance trail for the MCP request that initiated the AOR run. Re-running would risk duplicate artifacts, so the only safe response is a clean system error with the AOR audit reference in error details when available.

**Sequence for workflow.invoke_bounded:**
```
1. MCP preflight validates mode, allowlist, manifest, role card, ceiling, inputs, and writeback scope
2. For non-dry-run, MCP blocks if the predicted Operator-Briefs output already exists -> workflow_output_already_exists, aor_invoked=false
3. Handler calls AOR run_workflow()                  -> AOR writes/returns its own audit_id
4. Handler returns bounded response + audit metadata -> no MCPAuditLogger call from handler
5. Envelope writes MCP audit record                  -> if fails: system_error(workflow_invocation_audit_failed)
6. Return success only after MCP audit write succeeds
```

**Reconciliation rule:** A successful MCP response includes `aor_audit_id`, `audit_reconciliation`, `output_artifacts`, and `files_written`. Operators reconcile the AOR record by matching `aor_audit_id` to the AOR filename suffix in `07_LOGS/Agent-Activity/`, then reconcile the MCP request by matching `request_id` to the MCP filename suffix. If completion is ambiguous, do not retry blindly; inspect those records and any listed output artifact first.

---

### Prompt serving (handoff.runtime_draft_frame) — FAIL-OPEN

**Audit failure behavior:** Log the failure internally. Return the prompt response to the caller.

**Rationale:** Prompt serving is stateless — no vault reads, no writes (the prompt content is a template constant). An audit failure creates no governance risk. The caller receives the prompt regardless.

## Pre-Dispatch Failures (No Audit Record)

The following failure conditions occur before the audit system is reached. They produce no audit record:

| Failure | Why No Audit |
|---|---|
| Malformed request (JSON parse error) | No request_id; permission envelope not resolved |
| Config missing or invalid | No envelope; server cannot determine runtime_id or trust_tier |
| Surface not available in mode (mode violation) | This IS audited — mode check happens after envelope resolution |

**Clarification on mode violations:** A mode violation (requesting a tool in read_only mode) occurs after envelope resolution. The envelope is resolved successfully, but the surface is rejected. This rejection is audited with `outcome=error, error_code=surface_unavailable`.

---

## What Audit Records Do NOT Contain

The following are explicitly excluded from audit records:

- `proposed_content` — the full text of a staged proposal (this is in the staging artifact, not the audit record)
- `vault_file_content` — any raw content read from vault files during resource reads
- Credential, API key, or authentication data (there is none in V1 MCP — stdio local only)
- Full diff output (this is in `proposal.diff_preview` responses, not audit records)
- Full generated operator brief text from `workflow.invoke_bounded` (AOR artifacts hold content; MCP audit stores status and paths)
- The `current_sha256` or `proposed_sha256` of proposals (these are in the staging artifact)

Audit records describe what happened, not what content was read or proposed. The content lives in its respective artifact.

## MCPAuditLogger Implementation Contract (for Pass 4)

`audit/logger.py` must implement `MCPAuditLogger` with this interface:

```python
class MCPAuditLogger:
    def __init__(self, audit_dir: Path) -> None:
        """audit_dir = vault_root / '07_LOGS' / 'Agent-Activity'"""

    def log(
        self,
        request_id: str,
        surface_id: str,
        surface_class: str,          # "resource" | "tool" | "prompt"
        runtime_id: str,
        trust_tier: int,
        safety_mode: str,
        outcome: str,                # "success" | "error" | "partial"
        outcome_detail: str | None,
        files_read: list[str],
        files_written: list[str],
        error_code: str | None,
        error_message: str | None,
    ) -> None:
        """Write audit record to audit_dir.
        Raises MCPAuditError if write fails.
        Server/envelope decides fail-open vs fail-closed policy based on surface."""
```

**Invariants:**
- `log()` is always called with complete information — no partial records
- `log()` raises `MCPAuditError` on write failure — it does not swallow errors
- The server's outer envelope catches `MCPAuditError` and applies the surface-class policy
- `log()` never writes partial records (write-then-rename or similar atomicity guarantee)
- `MCPAuditLogger` has NO knowledge of fail-open vs fail-closed policy — that policy lives in `server.py`
- `MCPAuditLogger` is never called directly by resource, tool, prompt, or staging handlers

## Tightened Guardrail: Permitted Log Writes

The MCP server makes exactly three categories of MCP-owned non-canonical writes:

1. **Staging artifacts** — to `.chaseos/mcp-proposals/` only (via `proposal.submit` / `ProposalStore.stage()`)
2. **Audit records** — to `07_LOGS/Agent-Activity/` only (via `MCPAuditLogger`, called by the server/envelope)
3. **Approval request artifacts** — to `07_LOGS/Operator-Briefs/` only (via `approval_request.create`)

Active V2 `workflow.invoke_bounded` may cause AOR-owned Stage 7 writeback to `07_LOGS/Operator-Briefs/` through AOR only. The MCP handler does not write those files directly.

**Any MCP code path that writes to any other location is a guardrail violation** and must be caught in testing.

Specific tightening from the prior guardrail wording:
- MCP-owned `07_LOGS/Operator-Briefs/` writes are **only** from `approval_request.create`; AOR-owned Operator-Briefs writeback may occur only through `workflow.invoke_bounded` -> AOR
- `07_LOGS/Agent-Activity/` writes are **only** from `MCPAuditLogger.log()` when called by the server/envelope — not from any resource handler or tool directly
- `.chaseos/mcp-proposals/` writes are **only** from `ProposalStore.stage()` during `proposal.submit` — not from `approval_request.create` and not from any other module
- No MCP module writes to `00_HOME/`, `01_PROJECTS/`, `02_KNOWLEDGE/`, `03_INPUTS/`, `04_SOPS/`, `06_AGENTS/`, or any other top-level vault directory

---

## Audit Record Retention

Like the staging area, V1 has no automatic cleanup of audit records. `07_LOGS/Agent-Activity/` is the shared log directory for both AOR and MCP activity. Retention policy for the full directory is a vault-level concern, not a per-surface MCP concern.
