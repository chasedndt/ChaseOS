# ChaseOS Studio Phase 10 Implementation Tracker

This is the canonical Phase 10 implementation tracker for ChaseOS Studio until a stronger repo-native tracker replaces it. Agents should read this file before continuing Phase 10 work, then confirm the linked build logs for the pass they are about to touch.

## Current Shell Truth

- Primary product lane: `chaseos studio shell`.
- Primary implementation surface: `runtime/studio/shell/` native PyWebView shell.
- Legacy compatibility/QA harness: `chaseos studio desktop-shell-app`.
- Localhost routes remain useful for bounded browser/HTML/JSON evidence, but they are not the canonical Studio product shell.
- Direct packaged native visual screenshot QA is available through `chaseos studio packaged-app-visual-qa`; static/API evidence and labeled legacy-harness browser evidence must still say their own limitations honestly.
- Operator launch instructions are tracked in `06_AGENTS/ChaseOS-Studio-Launch-Instructions.md`: `chaseos studio shell` launches the native desktop app without a browser port, while `chaseos studio desktop-shell-app --host 127.0.0.1 --port <port>` launches the localhost compatibility harness on an operator-selected port.
- No co-development chat should edit shared Studio shell, CLI, architecture, log, daily, or index files without an explicit pass/file-boundary handoff.

## QA Safety Rule

Static QA must not call 10C write-surface APIs against the real repository vault.

After Phase 10C, methods such as `StudioAPI.create_node()` and `StudioAPI.create_link()` route through `StudioService` and can queue durable approval artifacts under `runtime/studio/approvals/`. Static QA over the real vault must inspect read-only surfaces only and must prove:

- no markdown writes,
- no Studio approval artifact writes,
- no provider or connector calls,
- no workflow execution,
- no Agent Bus writes,
- no graph/node/settings/canonical mutation.

Write-surface behavior belongs in isolated temp-vault tests such as `runtime/studio/shell/test_pass10c_write_surface.py`.
