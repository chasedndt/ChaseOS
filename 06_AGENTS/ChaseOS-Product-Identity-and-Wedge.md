# ChaseOS Product Identity and Wedge

## Product identity

**ChaseOS is a privacy-first agentic operating system where humans and AI agents work together through secure memory, knowledge graphs, automations, and permissioned execution.**

Expanded:

> ChaseOS turns scattered tools, files, workflows, agents, sources, runtimes, browser workflows, approvals, and outputs into one private command layer.

Plain English:

> ChaseOS helps AI-native operators run agents against real projects without losing context, control, privacy, or execution history.

Technical:

> ChaseOS is a privacy-first agentic operating system and control plane for source intelligence, structured project memory, knowledge graphs, runtime orchestration, bounded AOR workflows, governed writeback, approvals, browser/site workflow planning, and auditable agent execution.

Everyday:

> ChaseOS is where AI work stops being scattered. It keeps memory, project context, agent actions, approvals, and outputs connected in one private local-first command layer.

## Target wedge

The current wedge is **AI-native operators**: technical founders, solo builders, technical entrepreneurs, AI engineers, indie hackers, researchers, and creator-builders who already use multiple AI tools, agents, docs, repos, browser workflows, and runtimes, and are losing control of context, approvals, outputs, execution history, and runtime behavior.

Core pain:

- scattered work across ChatGPT, Claude, Codex, Claude Code, OpenClaw, Hermes, Obsidian, GitHub, terminals, browser tabs, Discord, docs, and notes;
- repeated re-briefing;
- lost project memory and decisions;
- no durable graph of sources, outputs, approvals, and actions;
- agents that can act without enough persistent operating context;
- browser/site workflows needing dry-run, approval, and audit;
- repeatable business processes that remain manual and ad hoc.

## What ChaseOS is not

ChaseOS must not be primarily framed as:

- an AI Startup Validation OS;
- a founder validation app;
- an Obsidian template;
- a second brain;
- a chatbot wrapper;
- a generic automation app;
- a finished autonomous company operator.

Startup validation is a mission pack inside the system, not the product identity.

## Founder Mode

**Founder Mode** is a ChaseOS operating mode for AI-native builders turning ideas, projects, research, launch assets, agents, and workflows into governed mission packs.

Placement:

```text
ChaseOS
└── Studio / Workspace modes
    └── Founder Mode / AI Builder Mode
        └── VentureOps mission packs
            └── Startup Validation & Launch
```

Founder Mode reuses existing ChaseOS substrate:

- Workspace Mode Layer `founder_venture` context;
- VentureOps workflow-pack and Mission Mode standards;
- SIC for source-grounded research and briefings;
- Graph substrate for project/source/output/provenance relationships;
- AOR for bounded workflow dispatch and writeback;
- SiteOps for dry-run site/landing-page workflow planning;
- Approval Center for review visibility;
- Gate and Permission Matrix for authority boundaries.

## Startup Validation & Launch

**Startup Validation & Launch** is the first Founder Mode / VentureOps mission pack.

It should turn one idea into:

- mission brief;
- venture gate score;
- pain research brief;
- customer/persona profile;
- competitor/problem map;
- positioning;
- landing page copy and local/static preview where supported;
- form spec;
- distribution posts;
- target prospecting plan;
- interview script;
- metrics tracker;
- approval preview;
- runtime handoff summary;
- decision report;
- graph links / Knowledge Box links where supported.

External actions remain proposal/approval-only unless a source-specific approved executor exists.

## Knowledge graph and Knowledge Boxes

ChaseOS should be explained as graph-first. Every project, source, agent, workflow, approval, decision, and output should become part of the governed ChaseOS graph where supported.

User-facing language: **Knowledge Boxes**.

A Knowledge Box is a UI-friendly graph container for a domain, project, customer, mission, or workflow family. It groups source nodes, project nodes, knowledge nodes, workflow nodes, runtime/agent nodes, approval nodes, output nodes, decision nodes, and provenance/runtime-handoff edges.

Example Founder Mode path:

```text
Startup Idea
→ Pain Hypothesis
→ Source Evidence
→ Customer Segment
→ Landing Page Draft
→ Form Spec
→ Launch Post
→ Approval Packet
→ Metrics Tracker
→ Decision Report
```

## How ChaseOS gets better with use

ChaseOS compounds because:

- source archives deepen;
- project state stays continuous;
- doctrine and decisions sharpen;
- runtime behavior becomes inspectable;
- workflow patterns become reusable;
- graph associations connect patterns across domains;
- failure-handling patterns accumulate.

## Current claims allowed when supported by repo truth

- local-first source intelligence and user-owned project memory;
- pluggable model/provider adapters;
- SIC source packages, workspaces, retrieval, structured outputs, and local persistence;
- graph/provenance substrate and Studio graph surfaces;
- AOR bounded workflows with writeback and audit;
- Workspace Mode Layer with `founder_venture` mode;
- VentureOps workflow-pack and Mission Mode substrate;
- SiteOps dry-run planning and bounded local browser proof lanes;
- Approval Center visibility/review surfaces;
- multi-runtime governance for OpenClaw, Hermes, Codex/Claude lanes;
- compounding operating context.

## Future or blocked claims

Future / roadmap:

- autonomous company operator;
- general computer-control runtime;
- arbitrary authenticated browser automation;
- generic Approval Center execution;
- live social posting, live deployment, CRM/payment mutation, or external sends;
- full self-governing OS.

Blocked unless explicitly approved and implemented:

- LinkedIn scraping;
- mass auto-DM;
- credential/session capture;
- unapproved provider/browser/connector calls;
- canonical graph/knowledge mutation outside approved paths;
- live external action without source-specific executor and approval.
