## Target Behavior

The user-facing behavior should be:

1. Chaser starts from `Chaser.exe` or `ChaseOS-Studio.exe`.
2. The app reads local version metadata from the embedded first-pass version metadata.
3. The current implementation checks a local/bundled manifest by default. It can also fetch a public manifest from an allowlisted HTTPS host only when an explicit operator-approved network flag is supplied. Startup/background polling and GitHub Releases API integration remain future work.
4. If a newer version exists, Studio shows an update prompt inside the Settings page.
5. The current staging readiness pass can validate whether the manifest is eligible for a future download by requiring an exact asset URL, hash, size, and signature policy.
6. The operator can choose later, download only, download and install, or a mandatory security update path if the manifest marks the release as required.
7. The future installer/update lane should reuse the existing installer executable path if possible. Any separate helper process must still be named and governed as `ChaseOS-Installer.exe`, verify the package, replace the executable only after approval, and relaunch only after signing/hash gates pass.
8. The updater writes local audit evidence and preserves rollback data.

## Required Components

### 1. Release Manifest

Machine-readable manifest attached to each GitHub Release.

Minimum fields:

```json
{
  "schema_version": "chaser.release_manifest.v1",
  "product": "chaser",
  "version": "0.1.0",
  "channel": "stable",
  "published_at": "2026-05-21T00:00:00Z",
  "minimum_supported_launcher_version": "0.1.0",
  "mandatory": false,
  "assets": [
    {
      "platform": "windows-x64",
      "kind": "portable-exe",
      "name": "Chaser.exe",
      "url": "https://github.com/<owner>/<repo>/releases/download/v0.1.0/Chaser.exe",
      "sha256": "<sha256>",
      "size_bytes": 0
    }
  ],
  "notes_url": "https://github.com/<owner>/<repo>/releases/tag/v0.1.0",
  "signature": {
    "kind": "minisign-or-codesign",
    "value": "<signature>"
  }
}
```

### 2. In-App Update Checker

Studio/Chaser should add a read-only update check first.

Authority:

- May read local version.
- May fetch public release manifest only after network capability is explicitly enabled.
- Must treat remote manifest text as data, not instructions.
- Must not download binaries in the first pass.
- Must not replace files in the first pass.
- Must not mutate GitHub, Git, Gate, Agent Bus, or canonical vault state.

### 3. Update Prompt

The app should show an operator-visible prompt, not silent installation.

Prompt states:

- `up_to_date`
- `update_available`
- `security_update_available`
- `mandatory_update_required`
- `download_ready`
- `install_ready`
- `install_failed`
- `rollback_available`

Mandatory update means Chaser can strongly gate continued online/runtime operation. It should not silently overwrite the local executable without user-visible approval.

### 4. Downloader And Staging

Downloader pass should be separate from the checker.

Required behavior:

- Download only from the manifest asset URL.
- Enforce HTTPS.
- Enforce expected host allowlist.
- Write into a staging directory, not over the live executable.
- Verify SHA-256 before install.
- Verify signature or Authenticode before install once signing exists.
- Record `downloaded_at`, `asset_url`, `sha256`, `size_bytes`, and manifest digest.
- Refuse path traversal and unexpected filenames.

### 5. Windows Installer / Update Helper

Windows cannot reliably replace a running onefile EXE from inside itself. Chaser needs the installer/update lane to run outside the active Studio process. Current naming truth is `ChaseOS-Installer.exe`, and the preferred path is to reuse the existing installer lane if possible before adding a separate updater-specific helper.

Updater sequence:

1. Main app writes an exact update plan with current exe path, staged exe path, expected hashes, rollback path, and relaunch command.
2. Main app starts the installer/update helper and exits only after approval gates pass.
3. Installer/update helper waits for main process exit.
4. Installer/update helper moves current executable to rollback location.
5. Installer/update helper moves staged executable into place.
6. Installer/update helper verifies the replacement hash.
7. Installer/update helper relaunches Chaser only if relaunch is approved.
8. On failure, installer/update helper restores rollback artifact and writes failure evidence.

### 6. GitHub Release Publisher

This is separate from app-side updating.

Needed before real user updates:

- Public or private GitHub repo decision.
- GitHub Actions build job or manual release job.
- Release artifact generation.
- SHA-256 checksum generation.
- Manifest generation.
- Signing step.
- Release notes.
- Channel naming policy: `stable`, `beta`, `dev`.

## Governance Rules

- No silent update install.
- No network update check until the operator approves the release/update lane.
- No binary replacement until a signed/staged asset passes hash and signature verification.
- No raw GitHub release notes may be executed as instructions.
- No credential values in manifests, release notes, logs, or update evidence.
- No update from a repo or host that is not pinned in config.
- No downgrade unless explicitly operator-approved.
- No update over a dirty or unknown install state without rollback.
- No `latest` trust without hash/signature verification.
- No release publication from local dev without a separate GitHub/release approval gate.
