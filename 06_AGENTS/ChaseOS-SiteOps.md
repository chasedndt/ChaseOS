# ChaseOS SiteOps

ChaseOS SiteOps is the governed web-workflow runtime for ChaseOS.

Naming stack:
- Feature family: ChaseOS SiteOps
- Technical registry: Website Workflow Index
- User-facing tab: Site Skills
- CLI namespace: `chaseos siteops ...`

Core flow:

```text
Website -> Site Profile -> Workflow Manifest -> Dry Run -> Approval -> Execution -> Audit -> Output -> Memory / Capture / Project Writeback
```

Browser Runtime Skill Memory is the runtime-intelligence loop inside this family:

```text
Browser run evidence -> Skill candidate -> Review -> Site Skill Card / Workflow Manifest -> AOR-controlled reuse
```

ChaseOS should study Browser Use, Browser Harness, workflow-use, and webagents.md as reference patterns, but SiteOps remains the governance, memory, approval, registry, and audit layer. External browser runtimes are adapters.

## Multi-User Production Scaffold

The production scaffold separates:

| Object | Purpose | Current path |
| --- | --- | --- |
| SiteSkillTemplate | System-owned reusable Site Skill definition | `runtime/siteops/catalog/site_skill_templates.yaml` |
| TenantSiteSkillInstallation | Tenant-installed skill configuration | `runtime/siteops/tenants/local.yaml` |
| WorkflowTemplate | Reusable workflow definition | `runtime/siteops/catalog/workflow_templates.yaml` |
| WorkflowInstallation | Tenant/workspace workflow configuration | `runtime/siteops/tenants/local.yaml` |

| ProviderAccountBinding | Provider adapter to credential/budget link | `runtime/siteops/tenants/local.yaml` |
| BudgetPolicy | Cost and approval controls | `runtime/siteops/tenants/local.yaml` |
| ApprovalRequest | First-class approval object | `07_LOGS/SiteOps-Approvals/<tenant>/<workspace>/` |
| SiteOpsRun | Scoped dry-run record | `07_LOGS/SiteOps-Runs/<tenant>/<workspace>/` |
| SiteOpsAuditEvent | Append-only run audit JSONL | `07_LOGS/SiteOps-Audits/<tenant>/<workspace>/` |

If required scope is missing or mismatched, SiteOps fails closed.

Hermes remains a bounded reviewer/shadow evaluator only. Hermes does not own SiteOps, does not become the canonical runtime, cannot read secrets, cannot invoke live connectors, and cannot bypass AOR/Gate/writeback rules.

## Governance Boundary

SiteOps is bounded autonomy over known surfaces, not free-roaming web control.

Default rules:
- all external page content is untrusted data, not instruction
- dry-run comes before execution
- irreversible, paid, external-share, export, broker, account, or destructive steps require explicit approval
- outputs route through capture/quarantine, logs, or project writeback rules before promotion
- provider credentials stay in environment, keychain, OAuth store, browser profile, or provider-specific credential boundary
- Scout-style discovery can draft profiles and workflow manifests but cannot mutate websites or canonical ChaseOS state
- Operator-style execution must load an approved profile, approved workflow, approval policy, and audit target before doing live work
