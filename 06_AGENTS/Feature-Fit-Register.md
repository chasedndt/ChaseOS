## Feature-Fit Triage Register

This register classifies all known and proposed ChaseOS features into the phase and layer where they best fit. It exists to prevent feature scope creep, ensure new ideas land in the right engineering phase, and provide a canonical answer when a feature request comes in and the correct home is unclear.

**This register is not a backlog.** It does not commit to or schedule any feature. It is a sorting table â€” operator uses it to decide where a feature should be spec'd and built.

**Governance:** Read this register before starting any new pass or feature proposal. If a feature is not listed, add it here first with a triage classification before implementing.

**Register supplement:** `[[ChaseOS-Feature-Family-and-Subfeature-Inventory]]` is the expanded inventory for repo-observed subfeatures and mini-features. Use it with this fit register before changing product docs or Studio navigation. The 2026-05-21 evidence matrix for the corrected pass is `[[docs/audits/2026-05-21_feature_family_deep_reconciliation]]`.

## Triage Rules

When a new feature request comes in, apply these rules in order:

1. **Does it move external content into quarantine?** â†’ Phase 8 (capture layer)
2. **Does it gather, inspect, normalize, trust-rank, or prepare operating inputs as source packs, evidence bundles, briefing inputs, action packets, or memory candidates before a workflow?** â†’ Phase 9 Acquisition + Normalization
3. **Does it work on promoted, quarantined content in a SIC workspace?** â†’ Phase 7 (SIC layer)
4. **Does it schedule, orchestrate, or automate execution?** â†’ Phase 9 (AOR layer)
5. **Does it present information or accept approvals from an operator?** â†’ Phase 10 (interface layer)
6. **Does it fit none of the above?** â†’ Flag as cross-cutting and do an architecture pass before implementing.

**Do not implement Phase 9 features in Phase 8 connectors.** Scheduling, background daemons, multi-step pipelines, and manifest-driven execution belong in AOR â€” not in individual connectors. Connectors are one-shot, operator-invoked tools.

**Do not implement Phase 10 surfaces in CLI commands.** The CLI is a scriptable operator interface, not a user-facing GUI. Rich interactive approval UX belongs in Phase 10.
